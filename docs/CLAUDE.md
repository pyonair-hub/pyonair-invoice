# CLAUDE.md — Documentation

Scope: end-user documentation for SolidInvoice, served by **Docusaurus** from this directory. This file layers on top of the project root `CLAUDE.md` (which covers code). When editing files under `docs/`, follow this guide.

## Audience & purpose

These docs are for **end users** of SolidInvoice — freelancers, small business owners, and self-hosters — not contributors or API consumers.

- Developer/contributor guidance lives in the project root (`README.md`, `CONTRIBUTING.md`, root `CLAUDE.md`).
- API reference is auto-generated and published at `https://solidinvoice.app/api/docs` — do not duplicate it here.

## Site structure

This is a Docusaurus site. Layout (paths relative to this `docs/` directory):

```
.                              # Docusaurus project root
├── docs/                      # actual doc content — every page lives here
│   ├── intro.mdx
│   ├── cron-job-setup.md      # standalone top-level page
│   └── <section>/
│       ├── _category_.json    # section label, position, index behaviour
│       └── <page>.md          # may also use index.md for the section landing
├── static/img/                # static assets (images, favicons)
├── src/                       # custom React components, CSS overrides
├── docusaurus.config.ts       # site config — don't edit unless adding plugins
└── sidebars.ts                # currently `autogenerated` from the docs/ tree
```

Key facts:

- `routeBasePath: '/'` — pages are served at the site root (no `/docs/` URL prefix).
- The sidebar is autogenerated from the folder tree under `docs/`. **Placing the file in the right folder is what makes it appear in nav** — there is no manual sidebar list to update.
- `onBrokenLinks: 'throw'` and `onBrokenMarkdownLinks: 'throw'` — broken links fail the build. Always verify links resolve before saving.

## Frontmatter

Every page must start with YAML frontmatter:

```yaml
---
title: Creating a client
description: Add a new client to SolidInvoice.
sidebar_position: 1
---
```

- `title` — used for the browser tab, navbar, breadcrumb, and (if no `# H1` is present) the rendered page heading. Sentence case, task-oriented ("Creating a client", not "Clients").
- `description` — one short sentence; used for SEO meta and search snippets.
- `sidebar_position` — order within the section. Pick the next available integer, or renumber siblings if inserting in the middle.
- Optional: `slug: /custom-url` if the file path produces the wrong URL. Leave off unless needed.

Use `.md` for plain pages. Use `.mdx` only when embedding React components (e.g. tabs).

## Sections (`_category_.json`)

A folder becomes a sidebar section when it contains a `_category_.json`:

```json
{
  "label": "Managing Clients",
  "position": 5,
  "link": {
    "type": "generated-index",
    "title": "Managing Clients",
    "description": "Add and manage your clients."
  }
}
```

- `label` — sidebar heading text.
- `position` — section order in the top-level sidebar.
- `link.type: "generated-index"` auto-builds an index page listing the section's pages. Alternative: `link.type: "doc"` + `link.id` to point at a hand-written index doc, or place an `index.md` inside the folder (see `installation-guide/`).

When creating a new section, copy `_category_.json` from a similar one and adjust `label` / `position`. Bump positions on subsequent sections to keep ordering explicit.

## Admonitions (callouts)

Use Docusaurus admonitions for side notes and warnings — never plain blockquotes or inline bold-warnings.

```markdown
:::info
Recurring tasks run automatically when using the single binary.
:::

:::tip
You can clone an existing client to speed up data entry.
:::

:::warning
Changing a client's currency after invoices exist is not supported.
:::

:::danger
This action permanently deletes all invoices for the client.
:::

:::note[Optional title]
Custom title goes in square brackets.
:::
```

Available types: `note`, `tip`, `info`, `warning`, `danger`.

## Images

Static assets live in `static/img/`. Group screenshots by section to keep things tidy:

```
static/img/managing-clients/create-client-form.png
```

Reference from a doc using a **root-absolute** path (not `../`):

```markdown
![The client create form](/img/managing-clients/create-client-form.png)
```

Always include meaningful alt text — used by screen readers and shown when the image fails to load. For a caption, follow the image with an italicized line:

```markdown
![The client create form](/img/managing-clients/create-client-form.png)
*The client create form, with required fields highlighted.*
```

Filename rules: lowercase, hyphenated, descriptive. Don't rename existing assets without searching for references — the build will fail on broken links.

## Links

- **Internal links** use relative `.md` paths so the build can verify them:
  `[Creating a company](../companies/creating-a-company.md)`
- **External links** use full URLs. Version-specific GitHub URLs should target `3.0.x` (the current dev branch); only link to `2.4.x` when documenting that version on purpose.
- Don't write bare URLs you haven't verified — broken links fail the build.

## Tabs (when needed)

For platform-specific instructions (Linux/macOS/Windows, Apache/Nginx, etc.), use Docusaurus tabs. Tabs require `.mdx`:

```mdx
import Tabs from '@theme/Tabs';
import TabItem from '@theme/TabItem';

<Tabs groupId="os">
  <TabItem value="linux" label="Linux">
    Linux instructions...
  </TabItem>
  <TabItem value="windows" label="Windows">
    Windows instructions...
  </TabItem>
</Tabs>
```

`groupId` syncs the chosen tab across pages. Only use tabs when the per-platform content differs meaningfully — otherwise a single section with a note is simpler.

## Voice & style

- Second person ("you"), task-oriented page titles ("Creating a company", "Switching between companies").
- Open with one sentence stating the goal, then jump into steps. No marketing intros, no restating the title.
- Use `## Heading` for top-level sections within a page; numbered lists for procedural steps.
- Inline UI elements in backticks: `+ Add Company`, `Create` button.
- Use admonitions for side notes and cautions, not bold inline warnings.

## Troubleshooting sections

Many pages end with a `## Troubleshooting` section. Format it consistently across the site so users can scan for their symptom:

- One `### subheading` per problem. The subheading is the **symptom or error message** the user will recognise — verbatim error strings go in backticks (`` `redirect_uri_mismatch` from Google ``), behavioural symptoms in plain prose (`Google buttons don't appear on the login page`).
- Follow each subheading with **1–2 short paragraphs** of body text explaining the cause and the fix. Do not use bullet lists or definition-list style (`**term** — explanation`) for the entries themselves — the subheading *is* the index, the paragraph is the answer.
- Keep entries focused on one symptom each. If a single root cause produces several symptoms, pick the most recognisable one as the subheading and mention the others in the body.
- Order entries by likelihood, not alphabetically. The most common problem goes first.
- Reference exact UI labels, env var names, and commands in backticks, the same as elsewhere on the page.

The [Troubleshooting section on the system installation page](./docs/installation-guide/system-installation.md) is the canonical example — match its shape on any new page that needs one.

## Building & verifying

Before claiming a doc change is complete, run a build to catch broken links and frontmatter errors:

```bash
bun run build
```

`bun run start` boots a dev server with hot reload for visual checks.

## Versioning

Current development branch: `3.0.x`. Some pages and the root README still link to `2.4.x` resources. When updating a page, check version-specific URLs (CONTRIBUTING link, install instructions, screenshots showing UI that has changed in 3.0). Flag mismatches rather than silently rewriting.

## Out of scope

This directory is for user-facing documentation only. Internal planning docs, TODOs, and engineering notes live in the root `todo/` directory, not here. Do not add scratch files, plans, or working notes under `docs/`.

## Checklist for a new page

1. Create the file under the correct section folder in `docs/docs/` (or top-level for a standalone page).
2. Add YAML frontmatter (`title`, `description`, `sidebar_position`).
3. Open with `# Title` matching the frontmatter title and a one-line purpose.
4. If creating a new section, add `_category_.json` and bump positions on later sections.
5. Place screenshots in `static/img/<section>/` and reference as `/img/<section>/<file>.png`.
6. Verify internal links are relative `.md` paths and resolve.
7. Run `bun run build` to catch broken links and frontmatter errors.
